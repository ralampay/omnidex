# Agents

This directory documents the agents currently shipped with omnidex.

## Available Agents

- `orchestrator`

## Agent Overview

- `orchestrator`: Runs as an interactive local-first chat agent backed by a GGUF
  model loaded through `llama-cpp-python`.

## CLI Entry Points

- `python -m omnidex`: Start the interactive orchestrator.
- `python -m omnidex --prompt "..."`: Run a single prompt and exit.

## Agent Creation and Tooling Primer

This section shows the smallest useful pieces for adding a new agent and a new
tool.

The goal is to keep the example simple:

- one basic tool that reads a directory
- one basic tool that summarizes that directory structure
- one simple agent that can use those tools

Read the sections in order. The examples start small and only come together at
the end.

### 1. Simple Agent Skeleton

Every agent should have:

- a class
- a `name`
- a `description`
- a `run(query, context="")` method

The smallest possible custom agent looks like this:

```python
from omnidex.agents.base import BaseAgent


class DirectoryAssistant(BaseAgent):
    name = "directory_assistant"
    description = "Helps inspect and summarize directories."

    def run(self, query: str, context: str = "") -> str:
        return f"You asked: {query}"
```

Purpose:

- `name` is what the orchestrator and logs use
- `description` helps routing and documentation
- `run(...)` is the main execution entry point

Typical location:

- `omnidex/agents/directory_assistant/agent.py`

### 2. Simple Tool Skeleton

All tools inherit from `BaseTool`.

A tool should define:

- `name`
- `output_fields`
- `run(...)`

Smallest example:

```python
from omnidex.tools.base import BaseTool


class EchoTool(BaseTool):
    name = "echo_tool"
    output_fields = ("content",)

    def run(self, text: str) -> dict[str, object]:
        return {
            "status": "ok",
            "content": text,
        }
```

Purpose:

- `name` is the planner-visible tool id
- `output_fields` declares the fields downstream steps should expect
- `run(...)` performs the domain work

Requirements for tools:

- inherit from `BaseTool`
- expose a stable `name`
- return structured dictionaries when possible
- provide a meaningful `content` field directly or indirectly
- keep `run(...)` inputs explicit so the planner can infer them from the
  function signature

Typical location:

- `omnidex/tools/<tool_name>.py`

### 3. A Real Simple Tool: Read A Directory

This tool reads a bounded recursive listing from a directory.

```python
from pathlib import Path

from omnidex.tools.base import BaseTool


class ReadDirectoryTreeTool(BaseTool):
    name = "read_directory_tree"
    output_fields = ("path", "tree", "content")

    def run(self, path: str) -> dict[str, object]:
        root = Path(path).expanduser()
        if not root.exists() or not root.is_dir():
            return {
                "status": "error",
                "path": str(root),
                "content": f"Error: directory not found: {root}",
            }

        lines: list[str] = []
        for entry in sorted(root.rglob("*"))[:200]:
            relative = entry.relative_to(root)
            prefix = "[D]" if entry.is_dir() else "[F]"
            lines.append(f"{prefix} {relative}")

        tree = "\n".join(lines)
        return {
            "status": "ok",
            "path": str(root),
            "tree": tree,
            "content": tree,
        }
```

Purpose:

- read local directory structure
- produce a reusable `tree` field
- still expose `content` for generic output handling

Typical location:

- `omnidex/tools/read_directory_tree.py`

### 4. Another Real Simple Tool: Summarize The Directory

This second tool summarizes the tree generated by the first tool.

```python
from omnidex.tools.base import BaseTool


class SummarizeDirectoryTreeTool(BaseTool):
    name = "summarize_directory_tree"
    output_fields = ("summary", "content")

    def run(self, tree: str) -> dict[str, object]:
        lines = [line for line in tree.splitlines() if line.strip()]
        dir_count = sum(1 for line in lines if line.startswith("[D]"))
        file_count = sum(1 for line in lines if line.startswith("[F]"))
        sample = "\n".join(lines[:12])
        summary = (
            f"Directory summary:\n"
            f"- {dir_count} directories\n"
            f"- {file_count} files\n\n"
            f"Top entries:\n{sample}"
        )
        return {
            "status": "ok",
            "summary": summary,
            "content": summary,
        }
```

Purpose:

- consume a specific upstream field
- produce a normalized summary
- return both a task-specific field and `content`

Typical location:

- `omnidex/tools/summarize_directory_tree.py`

### 5. Where Prompts Live And Why They Matter

Prompts should live with the agent or shared engine layer they belong to.

Typical prompt locations:

- agent-specific prompts:
  `omnidex/agents/<agent_name>/prompts.py`
- shared planner prompts:
  `omnidex/engine/planner_prompts.py`

An agent-level prompt file usually contains:

- the default system prompt for that agent
- any task-specific prompt builders used by internal services or model-backed
  tools

Simple example:

```python
DEFAULT_SYSTEM_PROMPT = (
    "You are a directory assistant. "
    "Be concise and accurate when describing local directory structure."
)
```

Purpose:

- define the role and behavior of the agent
- keep prompt text out of the main agent class
- make later prompt tuning easier

Typical location:

- `omnidex/agents/directory_assistant/prompts.py`

### 6. A Simple Agent With Tools

This version wires the tools into an agent. It does not use the planner yet.

```python
from omnidex.agents.base import BaseAgent
from omnidex.tools.create_output import CreateOutputTool


class DirectoryAssistant(BaseAgent):
    name = "directory_assistant"
    description = "Helps inspect and summarize directories."

    def __init__(self, **kwargs) -> None:
        super().__init__(
            tools=[
                ReadDirectoryTreeTool(),
                SummarizeDirectoryTreeTool(),
                CreateOutputTool(),
            ],
            **kwargs,
        )

    def run(self, query: str, context: str = "") -> str:
        reader = self.get_tool("read_directory_tree")
        summarizer = self.get_tool("summarize_directory_tree")
        output = self.get_tool("create_output")

        tree = reader.run(path=query)
        if tree.get("status") == "error":
            return str(tree.get("content") or "")

        summary = summarizer.run(tree=str(tree.get("tree") or ""))
        final = output.run(content=summary.get("content"))
        return str(final.get("content") or "")
```

Purpose:

- register tools on the agent
- let the agent retrieve them by `name`
- compose them in a fixed sequence

This is a good starting point when the workflow is deterministic.

### 7. When To Add Planner Support

Add planner support when:

- the agent has multiple possible tool chains
- the next tool depends on user intent
- the workflow is not a single fixed pipeline

The shared planner lives in:

- `omnidex/engine/planner.py`
- `omnidex/engine/planner_prompts.py`

That lets new agents reuse the same planning and execution infrastructure.

### 8. Putting It All Together

For a new simple agent, the usual directory layout is:

```text
omnidex/
  agents/
    directory_assistant/
      __init__.py
      agent.py
      prompts.py
  tools/
    read_directory_tree.py
    summarize_directory_tree.py
```

Suggested responsibilities:

- `agent.py`: agent class, tool registration, and top-level `run(...)` flow
- `prompts.py`: system prompt and any prompt builders specific to that agent
- `tools/*.py`: domain operations with stable `name`, `run(...)`, and
  `output_fields`

Minimal `__init__.py` example:

```python
from .agent import DirectoryAssistant

__all__ = ["DirectoryAssistant"]
```

If you want the orchestrator to delegate to the new agent, you then wire it into
the orchestrator's agent registry and routing descriptions.

Recommended build order:

1. create the tool classes first
2. test their return structures
3. add the agent skeleton
4. add the agent prompt file
5. wire the tools into the agent
6. only add planner support if the workflows are no longer deterministic
